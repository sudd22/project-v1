Cluster Architecture · RBAC
Grant a ServiceAccount read-only access to Pods in one namespace
~25%
Task

In namespace web, create a ServiceAccount named pod-reader-sa. Create a Role pod-reader that allows get, list, and watch on pods. Bind the Role to the ServiceAccount with a RoleBinding named read-pods. Finally, confirm the ServiceAccount can list pods but cannot delete them.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ k create namespace web namespace/web created root@controlplane:~$ k create serviceaccount pod-reader-sa -n web serviceaccount/pod-reader-sa created root@controlplane:~$ k create role pod-reader -n web \ --verb=get,list,watch --resource=pods role.rbac.authorization.k8s.io/pod-reader created root@controlplane:~$ k create rolebinding read-pods -n web \ --role=pod-reader --serviceaccount=web:pod-reader-sa rolebinding.rbac.authorization.k8s.io/read-pods created
Verify — terminal
root@controlplane:~verification
root@controlplane:~$ k auth can-i list pods -n web \ --as=system:serviceaccount:web:pod-reader-sa yes root@controlplane:~$ k auth can-i delete pods -n web \ --as=system:serviceaccount:web:pod-reader-sa no
Explanation — in plain words

RBAC in Kubernetes answers one question: "who can do what, where?" You always need three pieces glued together:

  • Subject — the identity. Here it is a ServiceAccount (an in-cluster identity used by pods/tools).
  • Role — the list of allowed verbs (get/list/watch) on resources (pods). A Role is namespaced; a ClusterRole is cluster-wide.
  • RoleBinding — the glue that says "this subject gets this Role in this namespace."

The --as=system:serviceaccount:<ns>:<name> flag lets you impersonate the account so you can test permissions without logging in as it. can-i returns a simple yes/no — the fastest way to prove your RBAC works.

Speed tip

Generating RBAC with kubectl create role/rolebinding imperatively is far faster than writing YAML. Use it on the exam unless the task forces a file.

Cluster Architecture · etcd
Take a snapshot backup of etcd
~25%
Task

Create a snapshot of the etcd database and save it to /opt/etcd-backup.db. The etcd certificates are in /etc/kubernetes/pki/etcd/ and the API server connects to etcd at https://127.0.0.1:2379.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ ETCDCTL_API=3 etcdctl snapshot save /opt/etcd-backup.db \ --endpoints=https://127.0.0.1:2379 \ --cacert=/etc/kubernetes/pki/etcd/ca.crt \ --cert=/etc/kubernetes/pki/etcd/server.crt \ --key=/etc/kubernetes/pki/etcd/server.key {"level":"info","msg":"created temporary db file","path":"/opt/etcd-backup.db.part"} {"level":"info","msg":"fetching snapshot","endpoint":"https://127.0.0.1:2379"} {"level":"info","msg":"saved","path":"/opt/etcd-backup.db"} Snapshot saved at /opt/etcd-backup.db root@controlplane:~$ # confirm the snapshot is valid root@controlplane:~$ ETCDCTL_API=3 etcdctl snapshot status /opt/etcd-backup.db --write-out=table +----------+----------+------------+------------+ | HASH | REVISION | TOTAL KEYS | TOTAL SIZE | +----------+----------+------------+------------+ | 7f3a1b2c | 61843 | 1297 | 5.2 MB | +----------+----------+------------+------------+
Explanation — in plain words

etcd is the brain of the cluster — every object (pods, secrets, configmaps) lives there as key-value data. Backing it up means you can rebuild the cluster state after a disaster.

etcd only talks over mutual TLS, so every command needs three files: the CA (--cacert) to trust the server, plus a client certificate and key (--cert, --key) to prove who you are. ETCDCTL_API=3 selects the v3 API where snapshot lives.

Common mistake

If you forget the cert flags you get context deadline exceeded — that is a TLS/auth failure, not a network outage. Always check the API server manifest at /etc/kubernetes/manifests/kube-apiserver.yaml for the exact cert paths.

Cluster Architecture · Node maintenance
Safely drain a node for maintenance, then bring it back
~25%
Task

Node node01 needs a kernel patch. Evict all workloads from it gracefully (ignoring DaemonSets and deleting any emptyDir pods), keep new pods from scheduling on it, then after maintenance mark it schedulable again.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ k drain node01 --ignore-daemonsets --delete-emptydir-data node/node01 cordoned Warning: ignoring DaemonSet-managed Pods: kube-system/kube-proxy-x8k2, kube-system/cilium-7t9q evicting pod default/web-7d9c8b6f5-2lq4r evicting pod default/web-7d9c8b6f5-9xk7p node/node01 drained root@controlplane:~$ k get nodes NAME STATUS ROLES AGE VERSION controlplane Ready control-plane 40d v1.36.0 node01 Ready,SchedulingDisabled <none> 40d v1.36.0 root@controlplane:~$ # ... maintenance happens ... root@controlplane:~$ k uncordon node01 node/node01 uncordoned
Explanation — in plain words

Three verbs control node scheduling:

  • cordon = "no new pods here" (marks node SchedulingDisabled) but leaves running pods alone.
  • drain = cordon plus politely evict the pods already running so they reschedule elsewhere.
  • uncordon = "open for business again."

--ignore-daemonsets is almost always required because DaemonSet pods (like kube-proxy) are meant to run on every node and cannot be evicted normally. --delete-emptydir-data acknowledges that pods using emptyDir will lose that scratch data.

Cluster Architecture · Static Pods
Create a static Pod that the kubelet manages directly
~25%
Task

On node01, create a static pod named web-static running image nginx:1.27. It must be managed by the kubelet (not the scheduler) so it restarts automatically even if the API server is down.

Solution — terminal
root@node01:~input / output
root@node01:~$ # find the static pod path the kubelet watches root@node01:~$ grep staticPodPath /var/lib/kubelet/config.yaml staticPodPath: /etc/kubernetes/manifests root@node01:~$ k run web-static --image=nginx:1.27 \ --dry-run=client -o yaml > /etc/kubernetes/manifests/web-static.yaml root@node01:~$ # kubelet auto-creates the pod within seconds — note the node suffix root@controlplane:~$ k get pods -o wide | grep web-static web-static-node01 1/1 Running 0 18s 10.244.1.7 node01
Explanation — in plain words

A static pod is defined by a YAML file dropped into the kubelet's manifest folder (usually /etc/kubernetes/manifests). The kubelet watches that folder and runs whatever it finds there — no scheduler, no API server needed. This is exactly how the control plane itself (apiserver, etcd, scheduler) runs.

Two giveaways that a pod is static: its name ends with -<nodename>, and you cannot kubectl delete it (it instantly comes back). To remove it, delete the manifest file.

Gotcha

The path comes from staticPodPath in /var/lib/kubelet/config.yaml. Always confirm it instead of assuming — some clusters customize it.

Workloads & Scheduling · Rollouts
Create a Deployment, scale it, update the image, then roll back
~15%
Task

Create a Deployment frontend with image nginx:1.25 and 3 replicas. Scale it to 5. Update the image to nginx:1.27. The new image is broken — roll back to the previous version.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ k create deployment frontend --image=nginx:1.25 --replicas=3 deployment.apps/frontend created root@controlplane:~$ k scale deployment frontend --replicas=5 deployment.apps/frontend scaled root@controlplane:~$ k set image deployment/frontend nginx=nginx:1.27 deployment.apps/frontend image updated root@controlplane:~$ k rollout status deployment/frontend Waiting for rollout to finish: 2 out of 5 new replicas have been updated... root@controlplane:~$ k rollout undo deployment/frontend deployment.apps/frontend rolled back root@controlplane:~$ k rollout history deployment/frontend REVISION CHANGE-CAUSE 2 <none> 3 <none>
Explanation — in plain words

A Deployment keeps a history of ReplicaSets. Each image change creates a new revision. rollout undo simply re-points traffic to the previous ReplicaSet — instant recovery, no YAML editing.

The container name matters in set image: nginx=nginx:1.27 means "container named nginx → new image." When you create a deployment imperatively, the container is named after the deployment's image, which is why it is nginx here. Check with k get deploy frontend -o jsonpath='{.spec.template.spec.containers[*].name}' if unsure.

Workloads & Scheduling · Config
Inject a ConfigMap into a Pod as both env vars and a mounted file
~15%
Task

Create a ConfigMap app-config with APP_COLOR=blue and APP_MODE=prod. Create a Pod app (image busybox:1.36, command sleep 3600) that reads APP_COLOR as an environment variable and mounts the whole ConfigMap at /etc/config.

Solution — terminal + YAML
root@controlplane:~input / output
root@controlplane:~$ k create configmap app-config \ --from-literal=APP_COLOR=blue --from-literal=APP_MODE=prod configmap/app-config created
app.yamlmanifest
apiVersion: v1 kind: Pod metadata: name: app spec: containers: - name: app image: busybox:1.36 command: ["sleep", "3600"] env: - name: APP_COLOR valueFrom: configMapKeyRef: name: app-config key: APP_COLOR volumeMounts: - name: cfg mountPath: /etc/config volumes: - name: cfg configMap: name: app-config
Verify — terminal
root@controlplane:~verification
root@controlplane:~$ k apply -f app.yaml && k exec app -- sh -c 'echo $APP_COLOR; ls /etc/config' blue APP_COLOR APP_MODE
Explanation — in plain words

A ConfigMap can be consumed two ways and this task uses both:

  • As an env var with configMapKeyRef — pulls one specific key into one variable.
  • As a volume — every key becomes a file (/etc/config/APP_COLOR, /etc/config/APP_MODE) whose contents are the value. Great for config files an app expects on disk.

The mount approach has a bonus: editing the ConfigMap updates the mounted files live (after a short delay), while env vars are fixed at pod start.

Workloads & Scheduling · Resources
Create a Pod with CPU/memory requests and limits
~15%
Task

Create a Pod limited (image nginx:1.27) that requests 64Mi memory and 250m CPU, and is limited to 128Mi memory and 500m CPU.

Solution — generate then edit
root@controlplane:~input / output
root@controlplane:~$ k run limited --image=nginx:1.27 $do > limited.yaml # $do expands to: --dry-run=client -o yaml
limited.yaml — resources blockmanifest
resources: requests: memory: "64Mi" cpu: "250m" limits: memory: "128Mi" cpu: "500m"
root@controlplane:~apply & verify
root@controlplane:~$ k apply -f limited.yaml pod/limited created root@controlplane:~$ k describe pod limited | grep -A4 Limits Limits: cpu: 500m memory: 128Mi Requests: cpu: 250m memory: 64Mi
Explanation — in plain words

Requests are what the scheduler reserves — it only places the pod on a node with at least that much free. Limits are the hard ceiling — exceed the memory limit and the container is OOMKilled; exceed the CPU limit and it is throttled (slowed), not killed.

Units: m = millicores, so 250m = ¼ of one CPU core. Mi = mebibytes (1024-based). There is no imperative flag for both requests and limits, so the exam-fast path is --dry-run to a file, then paste the resources block.

Services & Networking · Services
Expose a Deployment with a NodePort Service
~20%
Task

A Deployment web serves on container port 80. Expose it with a Service named web-svc of type NodePort on node port 30080, and verify it answers.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ k expose deployment web --name=web-svc \ --type=NodePort --port=80 --target-port=80 service/web-svc exposed root@controlplane:~$ # expose cannot pin the nodePort, so patch it to 30080 root@controlplane:~$ k patch svc web-svc -p '{"spec":{"ports":[{"port":80,"nodePort":30080}]}}' service/web-svc patched root@controlplane:~$ k get svc web-svc NAME TYPE CLUSTER-IP PORT(S) AGE web-svc NodePort 10.96.140.22 80:30080/TCP 12s root@controlplane:~$ curl -s node01:30080 | grep -o '<title>.*</title>' <title>Welcome to nginx!</title>
Explanation — in plain words

A NodePort Service opens the same high port (30000–32767) on every node and forwards it to your pods. Three port fields confuse people:

  • port — the Service's own port (cluster-internal).
  • targetPort — the container port traffic is sent to.
  • nodePort — the external port on each node.

kubectl expose is the quickest way to create the Service, but it cannot set a specific nodePort — so we patch it afterward. The Service finds its pods by copying the Deployment's label selector automatically.

Services & Networking · Ingress
Route two paths to two Services with one Ingress
~20%
Task

Create an Ingress shop on host shop.example.com that routes /cart to Service cart-svc:80 and / to Service web-svc:80. Use ingress class nginx.

Solution — YAML
shop-ingress.yamlmanifest
apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: shop spec: ingressClassName: nginx rules: - host: shop.example.com http: paths: - path: /cart pathType: Prefix backend: service: name: cart-svc port: number: 80 - path: / pathType: Prefix backend: service: name: web-svc port: number: 80
root@controlplane:~apply & verify
root@controlplane:~$ k apply -f shop-ingress.yaml ingress.networking.k8s.io/shop created root@controlplane:~$ k get ingress shop NAME CLASS HOSTS ADDRESS PORTS AGE shop nginx shop.example.com 192.168.49.2 80 20s
Explanation — in plain words

An Ingress is a set of routing rules for HTTP traffic. By itself it does nothing — an Ingress Controller (here NGINX) reads these rules and does the actual proxying. pathType: Prefix matches any URL that starts with the path, so /cart matches /cart/items too.

Order matters conceptually but the controller matches the longest/most specific prefix first, so /cart wins over / for cart URLs. The ADDRESS column only fills in once the controller assigns an IP — that confirms the controller picked up your rule.

Services & Networking · NetworkPolicy
Lock down a database so only the API tier can reach it
~20%
Task

In namespace prod, write a NetworkPolicy db-allow-api so pods labeled app=db only accept incoming traffic from pods labeled app=api, on TCP port 5432. All other ingress to the db must be denied.

Solution — YAML
db-policy.yamlmanifest
apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: db-allow-api namespace: prod spec: podSelector: matchLabels: app: db policyTypes: [Ingress] ingress: - from: - podSelector: matchLabels: app: api ports: - protocol: TCP port: 5432
Explanation — in plain words

A NetworkPolicy is a firewall for pods. The crucial rule: the moment a pod is selected by any policy, everything not explicitly allowed is denied. So this single policy both opens the api→db path and blocks everyone else — you do not need a separate "deny all" rule.

  • podSelector = which pods this policy protects (the db).
  • ingress.from = who is allowed in (the api pods).
  • ports = on which port.
Gotcha

NetworkPolicies need a CNI that enforces them (Calico, Cilium). With a plain flannel cluster the YAML applies but traffic is not actually blocked. The exam clusters use an enforcing CNI.

Services & Networking · DNS
Diagnose a pod that cannot resolve service names
~20%
Task

Pods report "name resolution failed" when reaching web-svc. Confirm whether cluster DNS is working and identify the cause.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ k get pods -n kube-system -l k8s-app=kube-dns NAME READY STATUS RESTARTS AGE coredns-76f75df574-2bkpq 0/1 CrashLoopBackOff 6 9m coredns-76f75df574-h9wzt 0/1 CrashLoopBackOff 6 9m root@controlplane:~$ k -n kube-system logs coredns-76f75df574-2bkpq plugin/loop: Loop ... detected "127.0.0.53:53" root@controlplane:~$ # test resolution from a throwaway pod root@controlplane:~$ k run dnstest --image=busybox:1.36 --rm -it --restart=Never \ -- nslookup web-svc.default.svc.cluster.local ;; connection timed out; no servers could be reached
Explanation — in plain words

Cluster DNS is provided by CoreDNS pods in kube-system. If they are not Running, every service-name lookup fails cluster-wide. The fix path is always: (1) check the CoreDNS pods are healthy, (2) read their logs, (3) test a lookup from a temporary pod.

The plugin/loop ... detected error means CoreDNS is forwarding to itself — usually because the node's /etc/resolv.conf points at a local stub (127.0.0.53). The fix is to point CoreDNS's forward at the real upstream resolver, or correct the kubelet's --resolv-conf. Naming format to remember: <service>.<namespace>.svc.cluster.local.

Storage · PV / PVC
Create a PersistentVolume, claim it, and mount it in a Pod
~10%
Task

Create a 1Gi hostPath PersistentVolume pv-data (path /mnt/data, access mode ReadWriteOnce). Create a PVC pvc-data requesting 500Mi. Mount it in Pod writer at /data.

Solution — YAML
storage.yamlmanifest
apiVersion: v1 kind: PersistentVolume metadata: {name: pv-data} spec: capacity: {storage: 1Gi} accessModes: [ReadWriteOnce] hostPath: {path: /mnt/data} --- apiVersion: v1 kind: PersistentVolumeClaim metadata: {name: pvc-data} spec: accessModes: [ReadWriteOnce] resources: {requests: {storage: 500Mi}}
root@controlplane:~verify binding
root@controlplane:~$ k apply -f storage.yaml && k get pvc pvc-data NAME STATUS VOLUME CAPACITY ACCESS MODES AGE pvc-data Bound pv-data 1Gi RWO 3s
Explanation — in plain words

Think of it as supply and demand. A PV is storage that exists (the supply). A PVC is a request for storage (the demand). Kubernetes binds a claim to any PV that is big enough and has matching access modes. The pod never references the PV directly — it references the PVC.

The claim asked for 500Mi and got bound to the 1Gi PV — claims bind to the smallest PV that satisfies them, and the pod gets the whole PV. ReadWriteOnce means one node can mount it read-write at a time.

Storage · StorageClass
Make a StorageClass the cluster default for dynamic provisioning
~10%
Task

A PVC fast is stuck Pending because no StorageClass is default. Mark the existing StorageClass local-path as the default so the claim provisions automatically.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ k get sc NAME PROVISIONER RECLAIMPOLICY DEFAULT local-path rancher.io/local-path Delete false root@controlplane:~$ k patch storageclass local-path -p \ '{"metadata":{"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}' storageclass.storage.k8s.io/local-path patched root@controlplane:~$ k get pvc fast NAME STATUS VOLUME CAPACITY ACCESS MODES AGE fast Bound pvc-9c1f...-local 2Gi RWO 5s
Explanation — in plain words

Dynamic provisioning means you don't hand-create PVs — a StorageClass + its provisioner creates them on demand whenever a PVC appears. A PVC with no storageClassName uses the cluster's default StorageClass.

The default is just an annotation: storageclass.kubernetes.io/is-default-class: "true". The instant you set it, the controller notices the pending claim and provisions a fresh PV — which is why fast jumps from Pending to Bound.

Troubleshooting · Pods
Fix a Pod stuck in ImagePullBackOff
~30%
Task

Pod api in namespace shop is stuck. Find the root cause and fix it so the pod runs.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ k -n shop get pod api NAME READY STATUS RESTARTS AGE api 0/1 ImagePullBackOff 0 2m root@controlplane:~$ k -n shop describe pod api | grep -A3 Events Failed to pull image "ngnix:1.27": not found Back-off pulling image "ngnix:1.27" root@controlplane:~$ # typo: "ngnix" -> "nginx". Fix the image in place. root@controlplane:~$ k -n shop set image pod/api api=nginx:1.27 pod/api image updated root@controlplane:~$ k -n shop get pod api NAME READY STATUS RESTARTS AGE api 1/1 Running 0 15s
Explanation — in plain words

ImagePullBackOff means the kubelet tried to download the container image and failed, so it is waiting before retrying. The cause is almost always one of: a typo in the image name, a wrong tag, or a private registry with no pull secret.

kubectl describe shows the exact failure in the Events section — here a misspelled ngnix. The golden troubleshooting rule: describe first, logs second. Describe shows scheduling/image/probe problems; logs show in-app crashes.

Note

You can edit a bare pod's image with set image, but most pod fields are immutable — if a field can't be changed live, delete and recreate from corrected YAML.

Troubleshooting · Nodes
Bring a NotReady node back online
~30%
Task

Node node01 shows NotReady and pods on it are stuck. Diagnose and restore the node.

Solution — terminal
root@node01:~input / output
root@controlplane:~$ k get nodes NAME STATUS ROLES AGE VERSION node01 NotReady <none> 40d v1.36.0 # SSH to the node and check the kubelet root@node01:~$ systemctl status kubelet ● kubelet.service - kubelet: The Kubernetes Node Agent Active: inactive (dead) root@node01:~$ systemctl enable --now kubelet root@node01:~$ systemctl is-active kubelet active root@controlplane:~$ k get nodes node01 NAME STATUS ROLES AGE VERSION node01 Ready <none> 40d v1.36.0
Explanation — in plain words

A node reports Ready only while its kubelet keeps sending heartbeats to the API server. If the kubelet is stopped, crashed, or misconfigured, the node flips to NotReady and the scheduler stops placing pods there.

The investigation order on the node itself: systemctl status kubelet → if dead, start it and check journalctl -u kubelet for why it died (bad config, expired certs, container runtime down). Here it was simply stopped, so enabling and starting it restores the heartbeat.

Also check

If the kubelet runs but the node is still NotReady, suspect the container runtime: systemctl status containerd. No runtime = no pods = NotReady.

Troubleshooting · Apps
Investigate a CrashLoopBackOff
~30%
Task

Pod worker keeps restarting with status CrashLoopBackOff. Find why and report the root cause.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ k get pod worker NAME READY STATUS RESTARTS AGE worker 0/1 CrashLoopBackOff 5 (40s ago) 3m root@controlplane:~$ # current container already died — read the PREVIOUS one's logs root@controlplane:~$ k logs worker --previous Error: cannot open config file /etc/app/config.yaml: No such file or directory process exited with code 1 root@controlplane:~$ k describe pod worker | grep -A5 Mounts Mounts: /etc/app from cfg (rw) root@controlplane:~$ k get configmap app-cfg Error from server (NotFound): configmaps "app-cfg" not found
Explanation — in plain words

CrashLoopBackOff means the container starts, crashes, and Kubernetes keeps restarting it with growing delays. The status tells you that it crashes — the logs tell you why.

The key trick: by the time you look, the running container has already died, so plain kubectl logs may be empty. Use --previous to read the logs of the container instance that just crashed. Here the app needs a config file that should come from a ConfigMap which doesn't exist — recreate app-cfg and the volume mount will populate /etc/app/config.yaml.

Troubleshooting · Control plane
Recover an API server that won't start after a manifest edit
~30%
Task

After editing /etc/kubernetes/manifests/kube-apiserver.yaml, kubectl commands fail with "connection refused." Find and fix the mistake.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ k get pods -A The connection to the server 127.0.0.1:6443 was refused root@controlplane:~$ # API down -> inspect the static pod via the container runtime root@controlplane:~$ crictl ps -a | grep apiserver a1b2c3 kube-apiserver Exited (1) 12s ago root@controlplane:~$ crictl logs a1b2c3 Error: unknown flag: --This-is-a-typo root@controlplane:~$ vim /etc/kubernetes/manifests/kube-apiserver.yaml # remove the bad flag line, save & quit root@controlplane:~$ # kubelet redeploys the static pod automatically; wait a moment root@controlplane:~$ k get pods -n kube-system | grep apiserver kube-apiserver-controlplane 1/1 Running 0 25s
Explanation — in plain words

The API server runs as a static pod, so a bad flag in its manifest makes it crash on startup — and because kubectl talks to the API server, you lose all kubectl access. You must debug below Kubernetes, directly at the container runtime with crictl.

Workflow: crictl ps -a to find the exited apiserver container → crictl logs to see the exact error → fix the manifest. The moment you save the file, the kubelet detects the change and recreates the pod. No systemctl restart needed for static pods.

Exam saver

Always cp /etc/kubernetes/manifests/kube-apiserver.yaml /root/apiserver.bak before editing. If you mangle it, restore the backup instantly.

Workloads & Scheduling · Taints
Taint a node and schedule a Pod that tolerates it
~15%
Task

Reserve node01 for GPU workloads: add taint gpu=true:NoSchedule. Then create a Pod cuda (image nginx:1.27) that has a matching toleration so it can still land there.

Solution — terminal + YAML
root@controlplane:~input / output
root@controlplane:~$ k taint node node01 gpu=true:NoSchedule node/node01 tainted
cuda.yaml — tolerations blockmanifest
spec: tolerations: - key: "gpu" operator: "Equal" value: "true" effect: "NoSchedule" containers: - name: cuda image: nginx:1.27
Explanation — in plain words

Taints repel; tolerations permit. A taint on a node says "keep pods away unless they explicitly tolerate me." The NoSchedule effect blocks new pods that lack the matching toleration. A pod's toleration must match the taint's key, value, and effect exactly.

Important nuance: a toleration allows placement but does not force it — the pod could still land on an untainted node. To pin it to the GPU node you would add nodeSelector or nodeAffinity too. The three effects are NoSchedule (block new), PreferNoSchedule (avoid if possible), and NoExecute (evict even running pods).

Workloads & Scheduling · Affinity
Pin a Pod to nodes with a specific label using nodeAffinity
~15%

Troubleshooting · NodePort access
NodePort works on one node but not another
~30%
Task

A NodePort Service responds when hitting node01 but times out via node02. Explain and confirm the cause.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ k get svc web-svc -o jsonpath='{.spec.externalTrafficPolicy}'; echo Local root@controlplane:~$ k get pods -o wide -l app=web NAME NODE web-abc node01 # all pods only on node01 root@controlplane:~$ # externalTrafficPolicy=Local only serves on nodes that host a pod. # node02 has no web pod -> times out. Troubleshooting · etcd full
API server read-only because etcd hit its space quota
~30%
Task

Writes fail cluster-wide with etcdserver: mvcc: database space exceeded. Reclaim space and clear the alarm.

Solution — terminal
root@controlplane:~input / output
root@controlplane:~$ ETCDCTL_API=3 etcdctl $E alarm list memberID:8e9e05c5 alarm:NOSPACE root@controlplane:~$ rev=$(ETCDCTL_API=3 etcdctl $E endpoint status -w json | grep -o '"revision":[0-9]*' | cut -d: -f2) root@controlplane:~$ ETCDCTL_API=3 etcdctl $E compact $rev compacted revision 61843 root@controlplane:~$ ETCDCTL_API=3 etcdctl $E defrag Finished defragmenting etcd member[127.0.0.1:2379] root@controlplane:~$ ETCDCTL_API=3 etcdctl $E alarm disarm memberID:8e9e05c5 alarm:NOSPACE (disarmed)
Explanation — in plain words

etcd keeps a history of every revision and has a default size quota (often 2GB). When it fills, etcd raises a NOSPACE alarm and goes read-only to protect itself — so all cluster writes fail.

The recovery is three steps: compact old revisions (drop history you don't need), defrag to actually return the freed space to the filesystem, then disarm the alarm so writes resume. Using $E (the cert flags from Q1) keeps the commands short.

Prevention

Schedule periodic auto-compaction (--auto-compaction-retention) so etcd never fills up in the first place.
