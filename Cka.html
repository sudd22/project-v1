What Is the CKA Certification?

The Certified Kubernetes Administrator (CKA) is a performance-based exam administered by the Cloud Native Computing Foundation (CNCF) in collaboration with the Linux Foundation. Unlike traditional multiple-choice certifications, the CKA exam requires you to solve real problems on a live, functioning Kubernetes cluster. You will be given a series of tasks and you must execute actual kubectl commands and create or modify real Kubernetes objects to earn points.

The CKA is widely regarded as one of the most valuable DevOps certifications available today. According to the CNCF annual report, certification exam enrollments reached over 291,000 in 2025, reflecting a 16% increase from the previous year. Kubernetes is the industry standard for container orchestration, and earning the CKA demonstrates that you possess the practical skills needed to deploy, manage, and troubleshoot Kubernetes clusters in production environments.

ls
The "list" command. It shows the contents (files and directories) of a location.

A flag that means "long format." Instead of just names, it shows detailed information: file permissions, number of links, owner, group, file size, modification date, and filename.

A flag that means "all." By default, ls hides files and directories whose names begin with a dot (.). The -a flag forces ls to show these hidden items as well. Hidden files often contain important configuration (like .kube/).

The target directory to list. This is where Kubernetes stores its cluster configuration files, including the admin kubeconfig, controller manager config, scheduler config, and PKI certificates.

cd

Text Processing — The Power Tools

Text processing commands are essential for the CKA exam. You will frequently need to search through logs, extract specific information, and pipe commands together. Mastering these tools will save you significant time during the exam.

grep
Stands for "Global Regular Expression Print." It searches for text patterns within files and outputs matching lines. This is your most-used troubleshooting tool.
-r
Recursive. Searches all files in the specified directory and all subdirectories, not just the top level.
-n
Show line numbers. Each matching line is prefixed with the filename and line number where the match was found, making it easy to locate the exact position.

The search pattern. grep will find any line containing the word "error." By default, the search is case-sensitive. Use -i to make it case-insensitive: grep -rni "error".

File Permissions

Linux uses a permission model to control who can read, write, and execute files. Each file has an owner, a group, and permission bits for three categories: owner, group, and others.

Terminal
ls -la /etc/kubernetes/admin.conf

The output shows something like: -rw------- 1 root root 5454 Jan 1 00:00 admin.conf

Understanding the Permission String
-
First character: file type. - means regular file. d means directory. l means symbolic link.
rw-------
Permission bits in three groups of three: rw- (owner), --- (group), --- (others). r = read, w = write, x = execute. - = permission denied. So here, only the owner (root) can read and write this file. Nobody else has any access.
root root
First "root" is the file owner. Second "root" is the group.

Changing Permissions with chmod

Terminal
chmod 600 /etc/kubernetes/admin.conf chmod +x /usr/local/bin/my-script.sh
Command Breakdown
chmod
Change mode — modifies file permissions.
600
Numeric (octal) permission representation. Each digit represents a category: first digit = owner, second = group, third = others. The values are: 4 = read, 2 = write, 1 = execute. You add them together: 6 = 4+2 (read+write), 0 = no permissions. So 600 means "owner can read and write, nobody else can access." Other common values: 644 (owner read/write, others read-only), 755 (owner full, others read+execute), 700 (owner full, nobody else).
+x
Symbolic mode. + adds a permission. x is the execute permission. This makes the file executable for everyone (owner, group, and others).

Changing Ownership with chown

Terminal
chown root:root /etc/kubernetes/pki/apiserver.crt
Command Breakdown
chown
Change ownership. Changes who owns a file and/or which group is associated with it.
root:root
The new owner and group, separated by a colon. root:root means owner = root, group = root. You could also use user:group format for any valid user and group on the system.

Killing Processes with kill

Terminal
kill 12345 kill -9 12345
Command Breakdown
kill
Sends a signal to a process, identified by its Process ID (PID). By default, it sends signal 15 (SIGTERM), which politely asks the process to terminate, giving it a chance to clean up.

The PID of the target process. You find PIDs using ps or top.

Systemd and Service Management

systemd is the init system and service manager on modern Linux distributions. It is responsible for starting and managing system services, including kubelet (the Kubernetes node agent). Understanding systemctl is essential for the CKA exam.

The command-line tool to control systemd. It manages services (start, stop, restart, enable, disable, check status).
status kubelet
Shows detailed status of the kubelet service: whether it is active (running), inactive (stopped), or failed; the PID; memory usage; recent log entries (last few lines of journal). This is usually the first command you run when troubleshooting node issues.
start kubelet
Starts the kubelet service if it is currently stopped.
stop kubelet
Stops the running kubelet service gracefully.
restart kubelet
Stops and then starts the kubelet. This applies configuration changes made to the kubelet configuration file.

Configures kubelet to start automatically at boot time. Without this, kubelet would not start after a system reboot.
disable kubelet
Removes the automatic start at boot. The service remains installed but will not start on its own.

Quickly checks if the service is currently running. Outputs "active" or "inactive" — useful in scripts.

journalctl -u kubelet -f --no-pager | tail -30

journalctl
The query tool for the systemd journal (system logs). It retrieves log messages from the system journal, which collects logs from all services managed by systemd.
-u kubelet
Filter logs to show only entries from the kubelet service unit. Without this, you would see logs from all system services.

Networking Commands

Kubernetes is fundamentally a networking system. Understanding Linux networking tools is essential for diagnosing connectivity issues.

Terminal
ip addr show ip route show ss -tlnp curl -k https://10.96.0.1:443/version nslookup kubernetes.default.svc.cluster.local
Command Breakdown
ip addr show
The ip command is the modern tool for network configuration (replacing the older ifconfig). addr show displays all IP addresses assigned to all network interfaces on the system. You will see interfaces like eth0, lo (loopback), and cni0 or flannel.1 (Kubernetes networking).
ip route show
Displays the kernel routing table — where network traffic goes for different destination IP ranges. Critical for understanding how pods route traffic between nodes.
ss
Socket statistics. Replaces the older netstat.
-t
Show TCP sockets only.
-l
Show only listening sockets (services waiting for incoming connections).
-n
Do not resolve service names — show numeric port numbers (e.g., "443" instead of "https").
-p
Show the process using each socket. This tells you which program is listening on which port. Requires root privileges.
curl
A tool for transferring data from or to a server using various protocols (HTTP, HTTPS, etc.). Used to test API endpoints.
-k
Allow insecure connections. Disables SSL certificate verification. Necessary when testing the Kubernetes API server which uses self-signed certificates.
https://10.96.0.1:443/version
10.96.0.1 is the default ClusterIP of the Kubernetes API server service. 443 is the HTTPS port. /version is a health endpoint that returns the Kubernetes version information. If this responds, the API server is healthy and reachable.
nslookup
DNS lookup tool. Queries DNS servers to resolve domain names to IP addresses.
kubernetes.default.svc.cluster.local
The fully qualified domain name (FQDN) of the Kubernetes API server service within the cluster's DNS system. If nslookup resolves this to an IP address, CoreDNS is functioning correctly.

Swap and Kernel Parameters

Critical Requirement

Kubernetes requires that swap is disabled on all nodes. If swap is enabled, kubelet will not start. The exam will have swap already disabled, but you must know how to check and disable it.

Terminal
sudo swapoff -a sudo sed -i '/ swap / s/^$$.*$$$/#\1/g' /etc/fstab
Command Breakdown
sudo
"Superuser do" — executes the command with root (administrator) privileges. Required for system-level changes.
swapoff -a
Disables all swap space immediately. -a means "all" — disables every swap device and file listed in /etc/fstab.
sed -i '/ swap / s/^$$.*$$$/#\1/g' /etc/fstab
This sed command comments out lines containing " swap " in /etc/fstab (the filesystem table that controls what gets mounted at boot). The # prefix comments out the line, preventing swap from being re-enabled after a reboot. The pattern ^$$.*$$$ captures the entire line and #\1 prepends a # to it.

Putting It All Together — Command Chaining

The real power of Linux comes from combining simple commands using pipes (|), redirection (>, >>), and subshells. Here are patterns you will use during the CKA exam:

Terminal
# Count total pods across all namespaces kubectl get pods --all-namespaces --no-headers | wc -l # Get all node names and their internal IPs kubectl get nodes -o wide --no-headers | awk '{print $1, $6}' # Find all pods in CrashLoopBackOff kubectl get pods --all-namespaces --no-headers | grep CrashLoopBackOff # Save output to a file kubectl get all --all-namespaces > /tmp/all-resources.txt # Append to a file (use >> not >) echo "# additional config" >> config.yaml
Command Breakdown
--no-headers
Removes the column header line from kubectl output. This is essential when piping output to tools like wc or awk, because the header line would be counted as data or produce incorrect results.
-o wide
Outputs additional columns beyond the default. For nodes, this adds the INTERNAL-IP, EXTERNAL-IP, OS-IMAGE, KERNEL-VERSION, CONTAINER-RUNTIME columns.
> (redirect)
Writes the command output to a file, overwriting any existing content. >> (double redirect) appends to the file instead of overwriting.
echo
Prints text to the terminal. When combined with >>, it appends text to a file.
PREREQUISITE

YAML Mastery

Every Kubernetes resource is defined using YAML. You must be fluent in reading, writing, and debugging YAML files.

What Is YAML?

YAML stands for "YAML Ain't Markup Language" (a recursive acronym). It is a human-readable data serialization format used to define configuration. Kubernetes uses YAML (and sometimes JSON) as the format for all resource definitions — pods, deployments, services, configmaps, and everything else. You will write YAML files for virtually every task on the CKA exam.

YAML Syntax Rules

YAML has a few fundamental rules that you must internalize:

  • Indentation matters: YAML uses spaces (NOT tabs) to indicate structure. The standard is 2 spaces per level. Incorrect indentation is the single most common source of YAML errors.
  • Case-sensitive: All keys and values are case-sensitive. kind: Pod is correct; kind: pod or Kind: Pod will cause errors.
  • Comments: Lines starting with # are comments and are ignored by the parser.
  • No tabs allowed: Using tab characters for indentation will cause a parse error. Always use spaces.

Key-Value Pairs

The most basic YAML structure is a key-value pair, separated by a colon and a space:

YAML
apiVersion: v1 kind: Pod metadata: name: my-pod namespace: default
YAML Breakdown
apiVersion: v1
apiVersion is the key. v1 is the value. They are separated by a colon followed by a space (: ). The space after the colon is mandatory. apiVersion tells Kubernetes which version of the API to use when interpreting this resource. v1 is the core API group (used for Pods, Services, ConfigMaps, Secrets, etc.).
kind: Pod
kind specifies the type of Kubernetes resource being defined. Pod means this YAML describes a Pod object. Other common kinds: Deployment, Service, ConfigMap, Secret, PersistentVolume.
metadata:
A key whose value is a nested object (notice the next lines are indented). metadata contains identifying information: the name, namespace, labels, and annotations of the resource.
  name: my-pod
Indented by 2 spaces, meaning it is a child of metadata. The resource name. Must be unique within the namespace for that resource type.
  namespace: default
The namespace in which to create this resource. If omitted, Kubernetes creates it in the default namespace (or whatever namespace is set in your kubeconfig context).

Lists (Arrays)

Lists in YAML are denoted by items prefixed with a hyphen and space (- ):

YAML

YAML Breakdown
containers:
A key whose value is a list (array). In Kubernetes, the containers field under spec is always a list because a pod can have one or more containers.
- name: nginx
The hyphen - starts a new list item. This item is an object with two keys: name and image. The hyphen must be followed by a space.
ports:
A nested list within the container definition. Each port entry is itself an object.
- containerPort: 80
A list item under ports. containerPort specifies the port number the container listens on. 80 is a number (no quotes), not a string.

Multi-Line Strings

Kubernetes YAML often requires multi-line text (for scripts, configuration files embedded in ConfigMaps, etc.). YAML provides two styles:

YAML — Literal Block Scalar
data: script.sh: | #!/bin/bash echo "Hello from the script" ls -la /tmp
YAML Breakdown
| (pipe)
The literal block scalar indicator. It preserves newlines exactly as written. Each line of the indented text becomes a separate line in the resulting string value. Use this when line breaks matter (like shell scripts or configuration files).
> (folded, not shown)
The folded block scalar. Replaces newlines with spaces, effectively joining lines into a single long string. Use this for long text paragraphs where line breaks should be treated as spaces.

Booleans, Numbers, and Null

YAML
enabled: true # Boolean true replicas: 3 # Integer cpu: "250m" # String (quotes prevent misinterpretation) restartPolicy: null # Null value (explicitly empty)
Common YAML Pitfall

Without quotes, YAML interprets yes, no, on, off, true, false as booleans, not strings. If you mean the literal string "yes", you must quote it: value: "yes". Similarly, values that look like numbers (e.g., a port number "8080") are fine without quotes because Kubernetes expects a number there, but a version string like "1.0" might be parsed as the number 1.0 — use quotes to be safe.

Containers Deep Dive

Kubernetes orchestrates containers. You must understand what containers are, how they work, and how to manage them at the command line.

What Are Containers?

A container is a lightweight, standalone, executable unit of software that packages application code together with all its dependencies (libraries, system tools, configuration files, and runtime). Containers share the host operating system's kernel but run in isolated user-space environments, giving each container its own file system, networking, process space, and resource limits.

Think of containers as more efficient alternatives to virtual machines. While a VM includes an entire operating system and runs on a hypervisor, a container shares the host kernel and uses Linux features like namespaces (for isolation) and cgroups (for resource limits) to achieve separation.

AspectVirtual MachineContainer

OverheadHigh (each VM runs its own OS)Low (shared kernel)
DensityTens per hostHundreds per host

Container Runtimes

A container runtime is the software responsible for running containers on a node. Kubernetes does not run containers directly — it delegates this to a container runtime through the Container Runtime Interface (CRI). The three main runtimes are:

  • containerd: The most widely used runtime in production Kubernetes clusters. It is a lightweight, high-performance runtime originally extracted from Docker. This is the recommended runtime for the CKA exam.
  • CRI-O: A lightweight runtime purpose-built for Kubernetes. It implements only the CRI specification and nothing more.
  • Docker: The original container runtime. Kubernetes deprecated direct Docker support (dockershim) in v1.24. However, Docker images are still fully compatible because Docker images follow the OCI (Open Container Initiative) standard, and containerd can run them.

Container Images

A container image is a read-only template that contains everything needed to run a container: the application code, runtime, libraries, environment variables, and default configuration. Images are stored in registries (like Docker Hub, Google Container Registry, or private registries) and are composed of multiple layers, where each layer represents a set of filesystem changes. Layers are shared between images to save disk space and speed up downloads.

Container Standards — OCI

The Open Container Initiative (OCI) defines industry standards for container image format and runtime. This ensures that images built with any tool (Docker, Buildah, Kaniko) can be run by any OCI-compliant runtime (containerd, CRI-O). All Kubernetes container runtimes must be OCI-compliant.

ARCHITECTURE

Kubernetes Architecture

Understanding how Kubernetes works internally is essential for both administration and troubleshooting. This chapter dissects every component.

Cluster Overview

A Kubernetes cluster consists of a set of machines called nodes. There are two types of nodes:

  • Control Plane (Master) Nodes: Run the components that manage the cluster — the API server, scheduler, controller manager, and etcd.
  • Worker Nodes: Run your actual application workloads (containers). Each worker node runs kubelet, kube-proxy, and a container runtime.

Control Plane Components

kube-apiserver

The API server is the front door to the Kubernetes cluster. Every single operation — whether it comes from a user running kubectl, from a kubelet reporting node status, or from a controller checking desired state — goes through the API server. It validates and processes RESTful requests, updates the state in etcd, and returns results to the caller.

The API server exposes a RESTful API over HTTPS (typically on port 6443). It handles authentication (who are you?), authorization (are you allowed?), and admission control (should this be modified/rejected?).

etcd

etcd is a distributed, consistent key-value store that serves as Kubernetes' single source of truth. Every piece of cluster state — nodes, pods, services, secrets, configmaps, deployments, and all other resources — is stored in etcd. If etcd loses data, you lose your cluster state. Therefore, regular etcd backups are critical.

etcd uses the Raft consensus algorithm to ensure that data is consistent across all etcd nodes in a high-availability setup. By default, etcd listens on port 2379 for client communication and port 2380 for peer communication between etcd cluster members.

kube-scheduler

When a new pod is created without a specified node, the scheduler is responsible for assigning it to a suitable node. It evaluates each node based on resource availability (CPU, memory), node selectors, affinity rules, taints and tolerations, topology constraints, and priority. It then selects the optimal node and writes the assignment to the pod's specification via the API server.

kube-controller-manager

The controller manager runs multiple controller loops, each responsible for a specific resource type. Each controller watches the API server for changes to its resource type and works to move the current state toward the desired state. Key controllers include:

  • ReplicaSet Controller: Ensures the correct number of pod replicas are running.
  • Deployment Controller: Manages ReplicaSets for Deployments, handling rollouts and rollbacks.
  • Node Controller: Monitors node health and handles node failures (marks nodes as NotReady, evicts pods).
  • Service Account Controller: Creates default ServiceAccounts for new namespaces.
  • Endpoint Controller: Populates Endpoints objects (maps services to pods).

Worker Node Components

kubelet

The kubelet is the primary node agent running on every worker node (and on control plane nodes for static pods). It receives pod specifications from the API server and ensures that the containers described in those specifications are running and healthy. It reports node and pod status back to the API server. The kubelet also:

  • Watches the /etc/kubernetes/manifests/ directory for static pod definitions.
  • Performs liveness and readiness probes on containers.
  • Manages volume mounts for pods.
  • Reports resource usage metrics.

kube-proxy

kube-proxy is a network proxy running on each node. It maintains network rules (iptables or IPVS rules) that allow network communication to reach pods from inside or outside the cluster. When you create a Kubernetes Service, kube-proxy programs the necessary rules to route traffic from the Service's virtual IP to the correct backend pods.

Container Runtime

The container runtime (containerd or CRI-O) is responsible for pulling container images, starting and stopping containers, and managing container lifecycle. The kubelet communicates with it through the CRI (Container Runtime Interface).

How a Pod Gets Created — The Full Journey

Understanding the pod creation lifecycle is essential for troubleshooting. Here is the step-by-step flow:

  1. You run kubectl apply -f pod.yaml. kubectl sends a POST request to the kube-apiserver.
  2. The API server authenticates your request, checks authorization (RBAC), and runs admission controllers.
  3. The API server validates the pod spec and stores the new Pod object in etcd (with status: Pending, no node assigned).
  4. The kube-scheduler watches for unscheduled pods. It detects the new pod and evaluates available nodes.
  5. The scheduler selects the best node and updates the pod's spec.nodeName field via the API server.
  6. The kubelet on the selected node detects the new pod assigned to it.
  7. The kubelet instructs the container runtime (containerd) to pull the image and start the containers.
  8. The container runtime pulls the image (if not already cached) and creates the containers.
  9. The kubelet starts the containers and monitors their health (probes).
  10. The kubelet reports the pod status back to the API server (Running, Succeeded, Failed, etc.).
  11. The API server updates etcd with the current status.

Role-Based Access Control (RBAC)

RBAC is a method of regulating access to resources based on the roles of individual users. In Kubernetes, RBAC is the primary authorization mechanism. It answers the question: "Is this user/service account allowed to perform this action on this resource?"

RBAC in Kubernetes has four key objects organized in two pairs:

ObjectScopePurpose
RoleNamespacedDefines a set of permissions (verbs on resources) within a single namespace
ClusterRoleCluster-wideDefines a set of permissions across the entire cluster (all namespaces, cluster-scoped resources)
RoleBindingNamespacedGrants the permissions defined in a Role or ClusterRole to users/groups/service accounts within a namespace
ClusterRoleBindingCluster-wideGrants the permissions defined in a ClusterRole to users/groups/service accounts across the entire cluster

Creating a Role

YAML — role.yaml
apiVersion: rbac.authorization.k8s.io/v1 kind: Role metadata: name: pod-reader namespace: default rules: - apiGroups: [""] # "" means the core API group resources: ["pods"] # Which resource types verbs: ["get", "list", "watch"] # Allowed actions
YAML Breakdown
apiVersion: rbac.authorization.k8s.io/v1
The API group and version for RBAC resources. All RBAC objects (Role, ClusterRole, RoleBinding, ClusterRoleBinding) use this API version.
kind: Role
This defines a namespaced Role. The permissions in this role only apply within the specified namespace (default here).
apiGroups: [""]
The empty string "" represents the core API group, which contains fundamental resources like pods, services, configmaps, secrets, and namespaces. Other API groups include "apps" (deployments, statefulsets), "batch" (jobs, cronjobs), "networking.k8s.io" (networkpolicies, ingress).
resources: ["pods"]
Specifies which Kubernetes resource types this role grants access to. Here, only pods.
verbs: ["get", "list", "watch"]
Specifies which operations are allowed. get = read a specific resource, list = list resources, watch = watch for changes. Other verbs include create, update, patch, delete.

Creating a RoleBinding

YAML — rolebinding.yaml
apiVersion: rbac.authorization.k8s.io/v1 kind: RoleBinding metadata: name: read-pods-binding namespace: default subjects: - kind: User name: jane apiGroup: rbac.authorization.k8s.io roleRef: kind: Role name: pod-reader apiGroup: rbac.authorization.k8s.io
YAML Breakdown
subjects
Defines who receives the permissions. A subject can be a User, Group, or ServiceAccount.
kind: User / name: jane
The user "jane" will receive the permissions. Note: Kubernetes does not manage users directly — it expects an external identity provider to authenticate users.
roleRef
References the Role or ClusterRole whose permissions are being granted. kind must be Role or ClusterRole, and name must match the exact name of that object.

ServiceAccounts

While Users represent humans, ServiceAccounts represent machine identities (pods, controllers, CI/CD pipelines). A pod can use a ServiceAccount to authenticate to the API server and perform actions.

Creates a ServiceAccount named my-sa in the current namespace.
kubectl create token my-sa --duration=3600s
Generates a bearer token for the ServiceAccount my-sa. --duration=3600s sets the token to expire after 3600 seconds (1 hour). This token can be used to authenticate to the API server.

Checking RBAC Access

Terminal

The action to test: can the subject list (verb) pods (resource)?

Tests the permission within the dev namespace. Roles are namespace-scoped, so the same user may have different permissions in different namespaces.

Installing Kubernetes with kubeadm

kubeadm is the official tool for bootstrapping a production-grade Kubernetes cluster. The exam tests your ability to create a cluster using kubeadm. The process involves preparing nodes, installing components, initializing the control plane, joining worker nodes, and installing a CNI plugin.

Step 1: Prepare All Nodes

Terminal — Run on ALL nodes (control plane + workers)
# Disable swap (required for kubelet) sudo swapoff -a # Load required kernel modules cat <
Command Breakdown
sudo swapoff -a
Disables all swap. Kubernetes requires swap to be off because swap makes resource accounting unreliable — kubelet cannot accurately track memory usage when the OS might swap container memory to disk.

This is a "here document" (heredoc). cat <<EOF reads everything until the line containing just EOF and outputs it. The pipe | sends that output to sudo tee, which writes the content to the specified file. tee is used because regular redirect > would not have sudo privileges.
overlay
The overlay filesystem module. Required by containerd for building container images using overlay/overlay2 storage driver, which layers filesystem changes efficiently.
br_netfilter
The bridge netfilter module. Required for Kubernetes networking — it enables iptables rules to process bridged network traffic, which is how pod-to-pod networking works across nodes.

net.bridge.bridge-nf-call-iptables = 1
Enables bridged network traffic to pass through iptables rules. Without this, kube-proxy's iptables rules would not work and pod networking would be broken.
net.bridge.bridge-nf-call-ip6tables = 1
Same as above but for IPv6 traffic.

Cluster Lifecycle Management

Upgrading a Kubernetes Cluster

Kubernetes clusters must be upgraded one minor version at a time (e.g., 1.35 to 1.36). The upgrade process involves upgrading the control plane first, then upgrading worker nodes one at a time.

Terminal — Control Plane Upgrade
# Upgrade kubeadm first sudo apt-get update sudo apt-get install -y --allow-change-held-packages kubeadm=1.36.0-* # Verify the upgrade plan sudo kubeadm upgrade plan # Apply the upgrade to the control plane sudo kubeadm upgrade apply v1.36.0 # Drain the control plane node (if it runs workloads) kubectl drain controlplane --ignore-daemonsets # Upgrade kubelet and kubectl sudo apt-get install -y --allow-change-held-packages kubelet=1.36.0-* kubectl=1.36.0-* sudo systemctl daemon-reload sudo systemctl restart kubelet # Uncordon the node kubectl uncordon controlplane
Command Breakdown
--allow-change-held-packages
Overrides the apt-mark hold restriction, allowing held packages to be upgraded. Required because we pinned these packages during installation.
kubeadm upgrade plan
Displays what will be upgraded without making any changes. Shows current and target versions for all control plane components. Always review the plan before proceeding.
kubeadm upgrade apply v1.36.0
Performs the actual upgrade of the control plane components: updates static pod manifests for API server, controller manager, scheduler, and etcd. Kubelet detects the new manifests and restarts the components.
kubectl drain controlplane --ignore-daemonsets

Extension Interfaces (CNI, CSI, CRI)

Kubernetes is extensible through standard interfaces:

  • CRI (Container Runtime Interface): Allows Kubernetes to use different container runtimes (containerd, CRI-O). Kubelet communicates with the runtime via the CRI gRPC protocol.
  • CNI (Container Network Interface): A plugin system for networking. CNI plugins (Flannel, Calico, Cilium, Weave) are responsible for allocating pod IP addresses, setting up network routes, and enabling pod-to-pod communication. When a pod is created, kubelet calls the CNI plugin to set up the pod's network namespace.
  • CSI (Container Storage Interface): A plugin system for storage. CSI drivers allow Kubernetes to provision and manage storage volumes from various backends (cloud providers like AWS EBS, GCE PD, Azure Disk; on-premises solutions like NFS, Ceph, or local storage).

Pods

A Pod is the smallest deployable unit in Kubernetes. It represents a group of one or more containers that share the same network namespace (they can communicate via localhost), the same IPC namespace, and optionally the same volumes. In practice, most pods contain a single container, but multi-container pods are used for patterns like sidecars (e.g., a log collector alongside the main app).

Pod Lifecycle Phases

PhaseDescription
PendingAccepted by the cluster but containers not yet running. Waiting for scheduling, image pulling, or volume attachment.
RunningAt least one container is running (or starting).
SucceededAll containers terminated with exit code 0 and will not be restarted (typical for Jobs).
FailedAll containers terminated, and at least one terminated with a non-zero exit code.
UnknownThe pod status cannot be obtained, typically due to a communication error with the node.

Init Containers

Init containers run before the main application containers start. They run to completion (exit code 0) sequentially — each init container must finish successfully before the next one starts. Common use cases: waiting for a service to be ready, performing database migrations, downloading configuration files.

YAML
apiVersion: v1 kind: Pod metadata: name: my-app-pod spec: initContainers: - name: wait-for-db image: busybox:1.36 command: ['sh', '-c', 'until nslookup db-service; do echo waiting; sleep 2; done'] containers: - name: app image: myapp:v1
YAML Breakdown
initContainers
A list of containers that run before the main containers list. They execute in order and must all complete successfully.
name: wait-for-db
The init container's name. This one waits for the database service to become reachable.
image: busybox:1.36
A minimal container image containing common Unix utilities. Useful for lightweight init containers that don't need a full application image.
command: ['sh', '-c', 'until nslookup db-service; do echo waiting; sleep 2; done']
A shell command that runs in a loop: it uses nslookup to check if db-service resolves in DNS. If it fails (service not ready), it prints "waiting" and sleeps 2 seconds before retrying. Once DNS resolves successfully, the command exits with code 0, allowing the next container to start.

Resource Requests and Limits

Kubernetes manages compute resources through requests and limits:

  • Request: The minimum amount of CPU/memory guaranteed to the container. The scheduler uses requests to decide which node to place the pod on.
  • Limit: The maximum amount of CPU/memory the container can use. If a container exceeds its memory limit, it gets killed (OOMKilled). If it exceeds its CPU limit, it gets throttled.
YAML
resources: requests: cpu: "250m" # 0.25 CPU cores (250 millicores) memory: "128Mi" # 128 mebibytes limits: cpu: "500m" # 0.5 CPU cores max memory: "256Mi" # 256 mebibytes max
Resource Units
250m
"250 millicores" = 0.25 CPU cores. The m suffix means "millicores" (1/1000 of a CPU core). 1000m = 1 full CPU core. 500m = half a CPU core.
128Mi
128 mebibytes of memory. Mi = mebibytes (1024 × 1024 bytes). M = megabytes (1000 × 1000 bytes). Gi = gibibytes. G = gigabytes. Always use Mi and Gi in Kubernetes for consistency.

Quality of Service (QoS) Classes

QoS ClassConditionBehavior on Resource Pressure
GuaranteedEvery container has requests = limits (for both CPU and memory)Last to be evicted
BurstableAt least one container has requests < limitsEvicted before Guaranteed pods
BestEffortNo requests or limits set at allFirst to be evicted

Pod Scheduling

Kubernetes provides several mechanisms to influence which node a pod gets scheduled on:

nodeSelector

YAML
spec: nodeSelector: disktype: ssd zone: us-east-1a

The simplest scheduling constraint. The pod will only be scheduled on nodes that have ALL the specified labels. First, label a node: kubectl label node node01 disktype=ssd zone=us-east-1a.

Node Affinity

YAML
affinity: nodeAffinity: requiredDuringSchedulingIgnoredDuringExecution: nodeSelectorTerms: - matchExpressions: - key: disktype operator: In values: - ssd - nvme preferredDuringSchedulingIgnoredDuringExecution: - weight: 80 preference: matchExpressions: - key: zone operator: In values: - us-east-1a
YAML Breakdown
requiredDuringSchedulingIgnoredDuringExecution
A hard requirement. The pod will NOT be scheduled unless a matching node is found. "IgnoredDuringExecution" means if the node labels change after scheduling, the pod continues running (it is not evicted).
operator: In
The node label's value must be one of the specified values. Other operators: NotIn, Exists, DoesNotExist, Gt, Lt.
preferredDuringSchedulingIgnoredDuringExecution
A soft preference with a weight (1-100). The scheduler prefers matching nodes but will schedule elsewhere if necessary.

Taints and Tolerations

While node affinity attracts pods to specific nodes, taints repel pods from nodes. A tainted node will not accept any pod unless the pod has a matching toleration.

Terminal

Command Breakdown
kubectl taint nodes node01
Applies a taint to node node01.
dedicated=gpu:NoSchedule
The taint in key=value:effect format. dedicated is the key, gpu is the value, NoSchedule is the effect. The three effects are: NoSchedule (new pods without toleration are not scheduled), PreferNoSchedule (soft — avoid scheduling if possible), NoExecute (evict existing pods without toleration AND prevent new ones).
dedicated=gpu:NoSchedule-
The trailing - removes the taint.

The Kubernetes Networking Model

Kubernetes imposes three fundamental networking rules:

  1. Every pod gets its own unique IP address.
  2. All pods can communicate with all other pods without NAT (Network Address Translation).
  3. Agents on a node (kubelet, system daemons) can communicate with all pods on that node.

This flat networking model is implemented by the CNI plugin (Flannel, Calico, Cilium, etc.). Each node gets a subnet (e.g., 10.244.1.0/24), and pods on that node get IPs from that subnet. Cross-node communication happens through overlay networking or direct routing.

Services

Pods are ephemeral — they can be created, destroyed, and replaced at any time. Each time a pod restarts, it may get a different IP address. A Service provides a stable network endpoint (virtual IP + DNS name) that routes traffic to a set of pods, regardless of pod IP changes.

ClusterIP Service (Default)

YAML
apiVersion: v1 kind: Service metadata: name: my-service spec: type: ClusterIP selector: app: my-app ports: - protocol: TCP port: 80 # Service port targetPort: 8080 # Container port
YAML Breakdown
type: ClusterIP
The default service type. Assigns a virtual IP address accessible only from within the cluster. External traffic cannot reach a ClusterIP service directly.
selector: app: my-app
Defines which pods this service routes to. The service matches pods that have the label app=my-app. Whenever a pod with that label exists and passes readiness checks, it receives traffic from the service.
port: 80
The port that the service itself listens on. Other pods connect to the service using <service-name>:<port> (e.g., my-service:80).
targetPort: 8080
The port on the container that traffic is forwarded to. The service receives traffic on port 80 and forwards it to port 8080 on matching pods.

NodePort Service

YAML
apiVersion: v1 kind: Service metadata: name: my-nodeport spec: type: NodePort selector: app: my-app ports: - port: 80 targetPort: 8080 nodePort: 30080 # Exposed on every node's IP
YAML Breakdown
type: NodePort
Exposes the service on a static port (30000–32767) on every node's IP address. Traffic sent to <NodeIP>:30080 from outside the cluster is routed to the service, then to matching pods. NodePort services also have an internal ClusterIP.
nodePort: 30080
The specific port number on the nodes. If omitted, Kubernetes assigns a random available port from the 30000-32767 range.

Command Breakdown

Creates a service that routes traffic to the pods managed by deployment my-deploy. The selector is automatically set to match the deployment's pod template labels.
--port=80 --target-port=8080

--type=NodePort
Sets the service type to NodePort instead of the default ClusterIP.

Ingress

An Ingress exposes HTTP/HTTPS routes from outside the cluster to services within the cluster. It provides URL-based routing, host-based routing, and TLS termination. An Ingress requires an Ingress Controller (like NGINX Ingress Controller) to function.

YAML
apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: my-ingress annotations: nginx.ingress.kubernetes.io/rewrite-target: / spec: ingressClassName: nginx rules: - host: app.example.com http: paths: - path: /api pathType: Prefix backend: service: name: api-service port: number: 80 - path: / pathType: Prefix backend: service: name: frontend-service port: number: 80 tls: - hosts: - app.example.com secretName: tls-secret
YAML Breakdown
ingressClassName: nginx
Specifies which Ingress Controller should handle this Ingress. Required when multiple controllers are installed.
host: app.example.com
Routes traffic only when the HTTP Host header matches this domain.
path: /api / pathType: Prefix
Routes requests whose URL starts with /api to the api-service. Prefix means it matches any URL that begins with the specified path (/api, /api/v1, /api/users, etc.). Exact would only match the exact path.
tls / secretName: tls-secret
Enables TLS (HTTPS) for this host. The TLS certificate and key are stored in the Kubernetes Secret named tls-secret.

Gateway API

The Gateway API is the modern evolution of Ingress, providing more expressive routing, traffic splitting, header-based matching, and request manipulation. It uses three main resources:

  • GatewayClass: Defines a type of gateway (like an IngressClass). Cluster-scoped.
  • Gateway: Defines an actual gateway instance (listening on specific ports, attached to GatewayClass).
  • HTTPRoute: Defines routing rules (which paths/hosts go to which services).
YAML — HTTPRoute
apiVersion: gateway.networking.k8s.io/v1 kind: HTTPRoute metadata: name: my-route spec: parentRefs: - name: my-gateway hostnames: - app.example.com rules: - matches: - path: type: PathPrefix value: /api backendRefs: - name: api-service port: 80

Network Policies

Network Policies control traffic flow at the IP address or port level for pods. They act as a firewall — by default, all pods can communicate with all other pods. Network Policies let you restrict this.

Important

Network Policies require a CNI plugin that supports them (Calico, Cilium, Weave). Flannel does NOT support Network Policies. If no Network Policy controller is installed, policies are accepted but not enforced.

Default Deny All Ingress

YAML
apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: default-deny-ingress namespace: production spec: podSelector: {} # Selects ALL pods in the namespace policyTypes: - Ingress
YAML Breakdown
podSelector: {}
An empty selector matches ALL pods in the namespace. This policy applies to every pod.
policyTypes: [Ingress]
This policy controls ingress (incoming) traffic. Since no ingress rules are defined, ALL incoming traffic is denied. Any pod wanting to receive traffic will need a specific allow rule.

Allow Specific Ingress Traffic

YAML
apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: allow-api-to-db spec: podSelector: matchLabels: app: database policyTypes: - Ingress ingress: - from: - podSelector: matchLabels: app: api ports: - protocol: TCP port: 5432
YAML Breakdown
podSelector / app: database
This policy applies to pods labeled app=database. Only these pods will have their ingress traffic filtered.
from / podSelector / app: api
Only allow ingress traffic from pods labeled app=api. All other pods are blocked.
port: 5432
Only allow traffic on TCP port 5432 (the default PostgreSQL port). Traffic on other ports from the api pods would also be blocked.

Namespace-based Selector

YAML
ingress: - from: - namespaceSelector: matchLabels: env: production

Allows traffic from all pods in namespaces labeled env=production.

This domain covers volumes, PersistentVolumes, PersistentVolumeClaims, StorageClasses, and dynamic provisioning.

Volumes

By default, a container's filesystem is ephemeral — when the container restarts, all data is lost. Volumes solve this by providing persistent or shared storage that survives container restarts.

emptyDir

An emptyDir volume is created when a pod is assigned to a node and exists as long as the pod is running. It starts empty and can be shared between containers in the same pod. It is deleted when the pod is removed.

YAML
spec: volumes: - name: shared-data emptyDir: {} containers: - name: writer volumeMounts: - name: shared-data mountPath: /data - name: reader volumeMounts: - name: shared-data mountPath: /shared
YAML Breakdown
emptyDir: {}
An empty volume that lives as long as the pod. The {} means no special configuration (defaults). The underlying storage is the node's filesystem.
volumeMounts / mountPath
Maps the volume to a directory inside the container. The writer container writes to /data, and the reader container reads from /shared — both pointing to the same underlying emptyDir volume.

hostPath

A hostPath volume mounts a file or directory from the host node's filesystem into the pod. This is useful for accessing node-level files (like logs or Docker socket) but creates a tight coupling to the specific node.

YAML
volumes: - name: host-logs hostPath: path: /var/log type: Directory

PersistentVolumes (PV) and PersistentVolumeClaims (PVC)

Kubernetes separates storage management into two objects:

  • PersistentVolume (PV): A piece of storage provisioned by an administrator or dynamically provisioned by a StorageClass. It represents actual storage capacity.
  • PersistentVolumeClaim (PVC): A request for storage by a user. It specifies size and access mode requirements. Kubernetes binds the PVC to a suitable PV.

PersistentVolume Example

YAML
apiVersion: v1 kind: PersistentVolume metadata: name: my-pv spec: capacity: storage: 10Gi accessModes: - ReadWriteOnce persistentVolumeReclaimPolicy: Retain storageClassName: standard hostPath: path: /mnt/data
YAML Breakdown
capacity / storage: 10Gi
The total size of this volume: 10 gibibytes (10 × 1024 MiB).
accessModes
ReadWriteOnce (RWO): Can be mounted as read-write by a single node. ReadOnlyMany (ROX): Can be mounted read-only by many nodes. ReadWriteMany (RWX): Can be mounted read-write by many nodes (not all storage backends support this).
persistentVolumeReclaimPolicy
What happens to the PV when the PVC is deleted. Retain: Keeps the data; the PV must be manually cleaned and reused. Delete: Deletes the PV and its backing storage. Recycle: (Deprecated) Wipes the data and makes the PV available again.
storageClassName: standard
Links the PV to a StorageClass. PVCs must request the same StorageClass to bind to this PV.

PersistentVolumeClaim Example

YAML
apiVersion: v1 kind: PersistentVolumeClaim metadata: name: my-pvc spec: accessModes: - ReadWriteOnce storageClassName: standard resources: requests: storage: 5Gi
YAML Breakdown
resources / requests / storage: 5Gi
Requesting 5 gibibytes. Kubernetes finds a PV with at least 5Gi available, matching StorageClass and access mode, and binds them together.

Using PVC in a Pod

YAML
spec: volumes: - name: data-vol persistentVolumeClaim: claimName: my-pvc containers: - volumeMounts: - name: data-vol mountPath: /var/lib/data

StorageClasses and Dynamic Provisioning

A StorageClass defines the "class" of storage — its provisioner, parameters, and reclaim policy. When a PVC references a StorageClass, Kubernetes dynamically provisions a PV using that class's provisioner. This eliminates the need for administrators to pre-create PVs.

YAML
apiVersion: storage.k8s.io/v1 kind: StorageClass metadata: name: fast-ssd provisioner: kubernetes.io/aws-ebs parameters: type: gp3 reclaimPolicy: Delete volumeBindingMode: WaitForFirstConsumer
YAML Breakdown
provisioner
The external provisioner that creates the actual storage volume. Different cloud providers and storage systems have their own provisioners.
parameters
Provisioner-specific configuration. For AWS EBS: type can be gp2, gp3, io1, etc.
volumeBindingMode: WaitForFirstConsumer
Delays PV provisioning until a pod using the PVC is actually scheduled. This ensures the PV is created in the same availability zone as the pod.
Exam Tip

For storage exam questions, the typical workflow is: (1) create a StorageClass or use an existing one, (2) create a PVC that references it, (3) mount the PVC in a pod. Use kubectl get pv and kubectl get pvc to verify binding status. If a PVC is stuck in Pending, check the StorageClass name, access mode match, and available capacity.

Troubleshooting

This is the highest-weighted domain at 30%. You must master systematic troubleshooting for control plane, nodes, applications, networking, and storage.

The Troubleshooting Methodology

Follow this systematic approach for every troubleshooting task:

  1. Identify the symptom: What is failing? A pod? A service? A node?
  2. Gather information: Use kubectl describe, kubectl logs, kubectl get events to collect details.
  3. Check the layer: Is it a scheduling issue? Container crash? Network problem? Storage problem?
  4. Isolate the cause: Use the information to pinpoint the specific misconfiguration or failure.
  5. Fix and verify: Apply the fix and verify that the issue is resolved.

StatefulSets

A StatefulSet is the workload object used for stateful applications — applications that need stable, unique network identifiers and persistent storage. Unlike Deployments where pods are interchangeable, StatefulSet pods have a stable identity that persists across rescheduling.

Key properties of StatefulSets:

  • Stable, unique network identifiers: Each pod gets a predictable name: <statefulset-name>-0, <statefulset-name>-1, <statefulset-name>-2, etc. Pods are created and terminated in strict order.
  • Stable, persistent storage: Each pod gets its own PersistentVolumeClaim that follows the pod even if it is rescheduled to a different node.
  • Ordered, graceful deployment and scaling: Pods are created sequentially (0, then 1, then 2). When scaling down, pods are terminated in reverse order (2, then 1, then 0).
  • Ordered, automated rolling updates: Updates are rolled out in reverse ordinal order by default.

StatefulSets require a headless Service to manage the network identity of the pods. The headless Service provides DNS entries for each pod.

YAML — Complete StatefulSet
apiVersion: v1 kind: Service metadata: name: nginx-headless spec: clusterIP: None selector: app: nginx-sts ports: - port: 80 --- apiVersion: apps/v1 kind: StatefulSet metadata: name: web spec: serviceName: "nginx-headless" replicas: 3 selector: matchLabels: app: nginx-sts template: metadata: labels: app: nginx-sts spec: containers: - name: nginx image: nginx:1.25 ports: - containerPort: 80 volumeMounts: - name: www-data mountPath: /usr/share/nginx/html volumeClaimTemplates: - metadata: name: www-data spec: accessModes: ["ReadWriteOnce"] storageClassName: "standard" resources: requests: storage: 1Gi
YAML Breakdown
serviceName: "nginx-headless"
Tells the StatefulSet which headless Service to use for pod DNS entries. Each pod gets a DNS name in the format: <pod-name>.<serviceName>.<namespace>.svc.cluster.local. For example: web-0.nginx-headless.default.svc.cluster.local.
replicas: 3
Creates three pods: web-0, web-1, web-2. They are created in this exact order, and each must be Running and Ready before the next one starts.
matchLabels / template / metadata / labels
The selector's matchLabels must exactly match the template's labels. This is how the StatefulSet identifies which pods it owns.
volumeClaimTemplates
This is what makes StatefulSets unique for storage. Instead of referencing a pre-existing PVC, you define a PVC template. For each replica, Kubernetes creates a NEW PVC from this template. When pod web-0 is rescheduled, it reattaches to the same PVC, preserving its data. The PVCs are named: www-data-web-0, www-data-web-1, www-data-web-2.
---
The YAML document separator. This single file contains two resources: a Service and a StatefulSet. The --- separates them so kubectl apply processes both.

DaemonSets

A DaemonSet ensures that a copy of a pod runs on every node (or a specific subset of nodes). When nodes are added to the cluster, DaemonSet pods are automatically started on them. When nodes are removed, the pods are garbage-collected.

Common use cases for DaemonSets:

  • Log collection: Running a log collector (like Fluentd or Filebeat) on every node to gather container logs.
  • Node monitoring: Running a monitoring agent (like Prometheus Node Exporter) on every node.
  • Networking: Running a CNI plugin agent (like Calico or Flannel) on every node.
  • kube-proxy: In many clusters, kube-proxy itself is deployed as a DaemonSet.
YAML — DaemonSet
apiVersion: apps/v1 kind: DaemonSet metadata: name: log-collector spec: selector: matchLabels: app: fluentd template: metadata: labels: app: fluentd spec: tolerations: - key: node-role.kubernetes.io/control-plane operator: Exists effect: NoSchedule containers: - name: fluentd image: fluentd:v1.16
YAML Breakdown
kind: DaemonSet
Ensures one pod per node. No replicas field — the count is automatically determined by the number of nodes.
tolerations / control-plane
Control plane nodes have a taint (node-role.kubernetes.io/control-plane:NoSchedule). Without this toleration, DaemonSet pods would only run on worker nodes. Adding this toleration ensures the DaemonSet also runs on control plane nodes.
operator: Exists
The Exists operator matches if the taint key exists, regardless of its value. This is a lenient match — any value of the key will be tolerated.

Jobs

A Job creates one or more pods that run a task to completion. Unlike Deployments that keep pods running indefinitely, Jobs are designed for finite tasks: batch processing, data migration, report generation, etc. The Job ensures the task completes successfully — if a pod fails, the Job restarts it.

YAML — Job
apiVersion: batch/v1 kind: Job metadata: name: pi-calculator spec: completions: 3 parallelism: 2 backoffLimit: 4 activeDeadlineSeconds: 100 template: spec: containers: - name: pi image: perl:5.34 command: ["perl", "-Mbignum=bpi", "-wle", "print bpi(2000)"] restartPolicy: Never
YAML Breakdown
apiVersion: batch/v1
Jobs belong to the batch API group.
completions: 3
The Job must successfully complete 3 pods total. The Job is considered finished only when 3 pods have completed with exit code 0.
parallelism: 2
Run up to 2 pods simultaneously. After one finishes, start the next one, always keeping 2 running until all 3 completions are achieved.
backoffLimit: 4
Allow up to 4 pod failures before considering the Job failed. Each failed pod is restarted.
activeDeadlineSeconds: 100
Maximum time the Job can run. After 100 seconds, the Job is terminated regardless of progress. This prevents runaway jobs.
restartPolicy: Never
Do not restart failed containers within a pod. Instead, create a new pod. The alternative is OnFailure, which restarts the container in the same pod. For Jobs, you must set this to Never or OnFailure — the default Always is not allowed.

CronJobs

A CronJob creates Jobs on a schedule, similar to the Linux cron utility. It is defined with a cron expression and a Job template.

YAML — CronJob
apiVersion: batch/v1 kind: CronJob metadata: name: daily-report spec: schedule: "0 2 * * *" successfulJobsHistoryLimit: 3 failedJobsHistoryLimit: 1 jobTemplate: spec: template: spec: containers: - name: report image: my-reporting-tool:latest restartPolicy: OnFailure
YAML Breakdown
schedule: "0 2 * * *"
Cron expression format: minute hour day-of-month month day-of-week. 0 2 * * * means "at 02:00 (2 AM) every day." The five fields are separated by spaces. An asterisk (*) means "every" for that field. Other examples: */5 * * * * = every 5 minutes, 0 0 * * 0 = midnight on Sundays, 30 8 1,15 * * = 8:30 AM on the 1st and 15th of every month.
successfulJobsHistoryLimit: 3
Keep the last 3 completed (successful) Job objects for reference. Older ones are automatically deleted.
failedJobsHistoryLimit: 1
Keep the last 1 failed Job object for debugging.
jobTemplate
The template used to create each Job. This is identical to a regular Job spec — it defines the pod that runs when the schedule triggers.
restartPolicy: OnFailure
If the container fails (non-zero exit code), restart it within the same pod. The alternative Never would create a new pod instead.

Liveness, Readiness, and Startup Probes

Probes are diagnostic checks performed periodically on containers. They determine whether a container is healthy and ready to receive traffic. Understanding probes is critical for the CKA exam because misconfigured probes are a common source of pod failures.

Liveness Probe

Determines whether a container is still running. If the liveness probe fails, kubelet kills the container and restarts it (subject to the pod's restartPolicy). Use liveness probes to detect when an application is deadlocked or stuck.

Readiness Probe

Determines whether a container is ready to accept traffic. If the readiness probe fails, the pod's IP is removed from all Service endpoints. The container is NOT restarted — it just stops receiving traffic until it becomes ready again. Use readiness probes for containers that need time to load data or warm up caches before serving requests.

Startup Probe

Determines whether a container has started successfully. While the startup probe is running, liveness and readiness probes are disabled. This is useful for containers with slow startup times — it prevents liveness probes from killing the container before it has finished initializing.

YAML — All Three Probe Types
spec: containers: - name: app image: myapp:v1 startupProbe: httpGet: path: /healthz port: 8080 failureThreshold: 30 periodSeconds: 10 livenessProbe: httpGet: path: /healthz port: 8080 initialDelaySeconds: 0 periodSeconds: 10 failureThreshold: 3 readinessProbe: httpGet: path: /ready port: 8080 periodSeconds: 5 failureThreshold: 3
YAML Breakdown
startupProbe / httpGet / path: /healthz
Performs an HTTP GET request to /healthz on port 8080. The probe passes if the HTTP response code is between 200 and 399.
failureThreshold: 30 / periodSeconds: 10
Try every 10 seconds, up to 30 attempts. This gives the application 300 seconds (5 minutes) to start. During this time, liveness and readiness probes are paused.
livenessProbe / failureThreshold: 3
After the startup probe succeeds, the liveness probe takes over. If it fails 3 consecutive times (3 × 10s = 30 seconds of failures), the container is killed and restarted.
readinessProbe / path: /ready
Checks a different endpoint (/ready). This could check whether the application has loaded its data and is ready to serve requests.

Security Contexts

A Security Context defines privilege and access control settings for a pod or container. It controls whether containers run as root, what Linux capabilities they have, and whether they can escalate privileges.

YAML — Pod Security Context
spec: securityContext: runAsUser: 1000 runAsGroup: 3000 fsGroup: 2000 containers: - name: app image: myapp:v1 securityContext: runAsNonRoot: true readOnlyRootFilesystem: true allowPrivilegeEscalation: false capabilities: drop: - ALL
YAML Breakdown
runAsUser: 1000
Runs the container processes as user ID 1000 (a non-root user). The default is 0 (root).
runAsGroup: 3000
The primary group ID for processes within the container.
fsGroup: 2000
Sets the group ownership of mounted volumes to group ID 2000. All processes in the container can read/write files on these volumes because they share the fsGroup.
runAsNonRoot: true
Prevents the container from running as root (UID 0). If the image's entrypoint tries to run as root, the pod will fail to start with an error.
readOnlyRootFilesystem: true
Makes the container's root filesystem read-only. The container cannot write to /, /tmp, /var, etc. Use emptyDir volumes for any writable directories the application needs.
allowPrivilegeEscalation: false
Prevents child processes from gaining more privileges than the parent. This is the default for non-root containers but should be explicitly set for security-sensitive workloads.
capabilities / drop: [ALL]
Drops all Linux capabilities from the container. Linux capabilities are fine-grained permissions (like NET_BIND_SERVICE for binding to ports below 1024, or SYS_ADMIN for system administration). Dropping ALL and only adding back what's needed follows the principle of least privilege.

ServiceAccounts and Pod Security

YAML — Assigning a ServiceAccount to a Pod
spec: serviceAccountName: my-sa automountServiceAccountToken: false
YAML Breakdown
serviceAccountName: my-sa
Assigns the ServiceAccount my-sa to this pod. The pod can then authenticate to the API server using this ServiceAccount's token and perform only the actions allowed by its RBAC bindings.
automountServiceAccountToken: false
Prevents the automatic mounting of the ServiceAccount token into the pod. By default, Kubernetes mounts a token file at /var/run/secrets/kubernetes.io/serviceaccount/token. If the pod doesn't need to communicate with the API server, disable this for better security — an attacker who compromises the pod won't have the token.

Troubleshooting Quick Flowchart

Pod Pending?

kubectl describe pod → Check Events. Look for: Insufficient resources, taint/toleration mismatch, node selector mismatch, unbound PVC.

Pod CrashLoopBackOff?

kubectl logs <pod> --previous → Check the previous container's logs for the crash reason. Also check kubectl describe pod for exit codes.

Pod ImagePullBackOff?

Check image name spelling, tag existence, registry accessibility. If private: create imagePullSecrets.

Service No Endpoints?

Service selector doesn't match pod labels. Compare kubectl describe svc Selector with kubectl get pods --show-labels.

Node NotReady?

SSH to node → systemctl status kubelet → journalctl -u kubelet. Check certificates, container runtime, network.

DNS Not Working?

kubectl get pods -n kube-system -l k8s-app=kube-dns → Check CoreDNS pods. Test: nslookup kubernetes.default from a pod.

PVC Stuck Pending?

kubectl describe pvc → Check StorageClass name match, access mode match, available PVs, and provisioner status.

RBAC Denied?

kubectl auth can-i <verb> <resource> --as=<user> → Test permissions directly. Check Role/ClusterRole rules and RoleBinding/ClusterRoleBinding subjects.
